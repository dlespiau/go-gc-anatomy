Invoking the Go compiler

go build invokes separate compiler and linker binaries behind the scenes as shown by the -x option:

$ go build -x
WORK=/tmp/go-build049312878
mkdir -p $WORK/_/home/damien/src/hello-world/_obj/
mkdir -p $WORK/_/home/damien/src/hello-world/_obj/exe/
cd /home/damien/src/hello-world
$GOROOT/pkg/tool/linux_amd64/compile -o $WORK/_/home/damien/src/hello-world.a -trimpath $WORK -p main -complete -buildid b41aca5027f01d4edf98e1676ee20bd7375510e3 -D _/home/damien/src/hello-world -I $WORK -pack ./main.go
cd .
$GOROOT/pkg/tool/linux_amd64/link -o $WORK/_/home/damien/src/hello-world/_obj/exe/a.out -L $WORK -extld=gcc -buildmode=exe -buildid=b41aca5027f01d4edf98e1676ee20bd7375510e3 $WORK/_/home/damien/src/hello-world.a
cp $WORK/_/home/damien/src/hello-world/_obj/exe/a.out hello-world

The Go compiler is really the compile tool. The compile tool can be invoked on its own:

$ go tool compile main.go
$ ls main*
main.go  main.o

Filesystem layout

While to the go binary is in $GOROOT/bin, tools are located in $GOROOT/pkg/tool/$GOOS_$GOARCH. On Linux with an x86â€‘64 processor, the compiler can be found in $GOROOT/pkg/tool/linux_amd64.

Compiler options

The compiler has quite a few options to play with. Most of them are to support the integration with go build. Some, however, can help understanding and debugging the compiler itself.

Consider the following Hello World program.

package main

func printHelloWorld() {
        println("Hello, World!")
}

func main() {
        printHelloWorld()
}

The -m option tells us about optimisation decisions the compiler has done. In this instance printHelloWorld can be inlined.

$ go tool compile -m main.go
main.go:3: can inline printHelloWorld
main.go:7: can inline main
main.go:8: inlining call to printHelloWorld
Compiler options can be passed from go build as well with the -gcflags parameter, eg. go build -gcflags '-m'.

When applicable, I’ll indicate how to get relevant debugging information for the topic discussed in the rest of this article.

Sources

The entry point of the compiler is in compile but the bulk of the code is in various internal packages starting by gc. The internal packages are split between compiler-specific ones in src/cmd/compile/internal and some used by several tools in src/cmd/internal. To list a few interesting ones:

gc

main package, AST, type checking, inlining, escape analysis, closure handling, some optimisations and lowering on the AST.

syntax

Lexical analysis, tokenization and building of the first AST representation.

ssa

The backend part of the compiler is an SSA IR with many optimisations done at that level.

obj

Machine code generation.

The journey of a go program

Overview

  1. Initialisation.

  2. Lexical analysis and parsing into the cmd/compile/internal/syntax AST.

  3. Translation into the cmd/compile/internal/gc AST.

  4. Some gc AST passes: type checking, escape analysis, inlining, closure handling, expression evaluation ordering (order.go), and some lowering and optimisation (walk.go).

  5. Translation into the cmd/compile/internal/ssa SSA form.

  6. Optimisation and lowering of SSA form.

  7. Translation from SSA form to assembler instructions.

  8. Translation from assembler instructions to machine code.

  9. Writing lots of output: machine code, DWARF symbols, type and reflection info, export data

Initialisation

The entry point of the compiler is in main.go but the interesting entry point is really Main in the gc package.

Various tasks are done there: creating built-in packages like unsafe, setting up the universe block (universe.go), parsing the command line options and, finally building the AST on which high-level passes are done.

Once the files are parsed, the global variable xtop contains the list of top level Nodes.

There are actually two different AST. One in the syntax package and one in the gc package. The former is transformed into the latter right at the start (in noder.go, parseFiles). It seems there’s a will to get rid of the gc AST in favour of the syntax one (source).

Of course, the reference to have handy when looking at the parsing code is the The Go Programming Language Specification.

Inlining

Inlining is quite an important optimisation. Not only it removes some function call overhead but, more importantly, it enables further function-wide optimisations at a later stage.

The inlining code is in inl.go with a nice explanatory comment at the top of the file. A first pass in caninl decides which functions can be inlined. A second pass, inlcalls, will expand inlinable functions at their call sites.

ishairylist and ishairy walk the function and decide if it’s too hairy (ie. complex) to be inlined or not with a budget of 80 AST nodes.

The Go compiler can inline across package boundaries.

The inliner only inlines leaf functions (with some exceptions). Mid-stack inlining is in the pipeline.

The -m flag can be used to see the inliner in action:

package main

import "fmt"

func add(a, b int) int {
        return a + b
}

func main() {
        fmt.Println("1 + 1 =", add(1, 1))
}
$ go tool compile -m inlined.go
inlined.go:5: can inline add
inlined.go:10: inlining call to add
inlined.go:10: "1 + 1 =" escapes to heap
inlined.go:10: add(1, 1) escapes to heap
inlined.go:10: main ... argument does not escape

The undocumented option -m=2 gives more insights into the reasons the compiler doesn’t inline certain functions.

$ cat not-inlined.go
package main

func dummy() {}

func ClosureCreation() {
        _ = func() {}
}

var x = []byte{0}

func RangeLoop() {
        for _ = range x {
        }
}

var ch chan struct{}

func Select() {
        select {
        case <-ch:
        default:
        }
}

func Go() {
        go dummy()
}

func Defer() {
        defer dummy()
}

func LocalTypeDecl() {
        type I int
}

func main() {
        ClosureCreation()
        RangeLoop()
        Select()
        Go()
        Defer()
        LocalTypeDecl()
}
$ go tool compile -m=2 not-inlined.go
not-inlined.go:3: can inline dummy as: func() {  }
not-inlined.go:5: cannot inline ClosureCreation: unhandled op CLOSURE
not-inlined.go:6: can inline ClosureCreation.func1 as: func() {  }
not-inlined.go:11: cannot inline RangeLoop: unhandled op range
not-inlined.go:18: cannot inline Select: unhandled op select
not-inlined.go:25: cannot inline Go: unhandled op PROC
not-inlined.go:29: cannot inline Defer: unhandled op defer
not-inlined.go:33: cannot inline LocalTypeDecl: unhandled op DCLTYPE
not-inlined.go:37: cannot inline main: non-leaf function
not-inlined.go:6: ClosureCreation func literal does not escape
